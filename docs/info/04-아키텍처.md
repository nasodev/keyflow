# 04. 아키텍처

> 이 문서는 **개발자**가 코드를 만지기 전에 "전체가 어떻게 묶여 있는지"를 30분 안에 이해하도록 하는 그림이다.

## 한 페이지 요약

```
┌──────────────────────────────────────────────────────────────────┐
│                  GameplayScene.unity (유일한 씬)                  │
│                                                                  │
│  ┌─────────┐  ┌────────┐  ┌──────────┐  ┌──────────┐           │
│  │ Start   │  │ Main   │  │ Gameplay │  │ Results  │  ← 4 화면 │
│  └─────────┘  └────────┘  └──────────┘  └──────────┘           │
│       └─────────┘                                                │
│   ScreenManager가 SetActive 토글                                 │
│                                                                  │
│  + 오버레이 3개: Settings / Pause / Calibration                  │
└──────────────────────────────────────────────────────────────────┘
                            ▲
                            │ 처음부터 통째 재생성
                            │
              ┌─────────────────────────────┐
              │  KeyFlow/Build W4 Scene     │
              │  (SceneBuilder.cs)          │
              │  → 모든 GameObject + 와이어  │
              └─────────────────────────────┘
```

핵심 원칙 4개:

1. **씬은 하나뿐이다** — `GameplayScene.unity`. 화면 전환은 SetActive 토글.
2. **씬은 코드로 짓는다** — `SceneBuilder.Build()`가 단일 진실 소스. 인스펙터에서 손으로 만지지 않는다.
3. **시간은 dspTime이 권위다** — `Time.time`이 아니라 `AudioSettings.dspTime`. 오디오와 노트가 같은 시계를 본다.
4. **테스트 시 의존성은 주입한다** — `SerializeField` 대신 `SetDependenciesForTest`.

## 시간 모델: dspTime이 권위다

```
[기기 시각] ────────────────► AudioSettings.dspTime (오디오 스레드)
                                         │
                                         ▼
                          AudioSyncManager.SongTimeMs
                          = (dspNow − songStartDsp − calibSec) * 1000
                                         │
                                         ▼
              ┌──────────────┬──────────┴──────────┬──────────────┐
              ▼              ▼                     ▼              ▼
      NoteSpawner      NoteController       JudgmentSystem    HoldTracker
      (2초 앞 스폰)    (위치 보간)          (delta 계산)      (Tick)
```

**왜 `Time.time`이 아닌가**: Unity의 `Time.time`은 메인 스레드 프레임 시각이라 오디오와 미세하게 어긋난다. 리듬게임에서는 ±10ms 오차도 판정에 영향을 준다. 오디오 출력은 `dspTime`을 기준으로 스케줄되므로, 노트도 같은 시계를 봐야 동기가 맞는다.

**테스트 seam**: `AudioSyncManager.TimeSource`(`ITimeSource`)는 EditMode 테스트에서 수동 시계를 주입할 수 있는 인터페이스다. 실제 오디오 스레드 없이 dspTime을 결정적으로 진행시킬 수 있다(`Assets/Scripts/Gameplay/AudioSyncManager.cs:6–11`).

        │
        ▼

        │
        ▼
ContinueAfterChartLoaded()
        │
        ├── HasCalibration? → 바로 BeginGameplay()
        └── 없으면        → CalibrationController.Begin(BeginGameplay)
        │
        ▼
BeginGameplay()
        │
        ├── audioSync.Stop()           ← 카운트다운 동안 NoteSpawner 잠재움
        ├── spawner.Initialize(diff)
        │
        ▼

        │
        ▼

        │

**왜 `audioSync.Stop()`을 두 번 부르는가**: 카운트다운이 3초간 진행되는데, `NoteSpawner.Update`는 `audioSync.IsPlaying`을 가드로 본다. 이전 세션의 stale `songStartDsp`로 `SongTimeMs`가 거대한 값이 되면 곧 떨어질 노트들이 카운트다운 중에 한꺼번에 스폰된다. SP11 카운트다운 도입 시 발견된 회귀 — 두 곳에서 `Stop()` 호출로 차단(`Assets/Scripts/Gameplay/GameplayController.cs:62, 95`).

## 캘리브레이션 흐름

```
첫 플레이 시 (UserPrefs.HasCalibration == false)
        │
        ▼
CalibrationController.Begin(onDone)
        │
        ├── Show()
        ├── promptText = "탭하세요"
        │
        ▼ (사용자 Start 버튼)
RunCalibration() 코루틴
        │
        ├── 8회 클릭을 dspTime + 0.5s 간격으로 PlayScheduled
        ├── 사용자 탭마다 AudioSettings.dspTime 기록
        │
        ▼
Evaluate()
        │
        ▼
CalibrationCalculator.Compute(expectedDspTimes, tapDspTimes)
        │
        ├── 첫·마지막 샘플 트림 (count >= 6일 때)
        ├── median offset 계산 (sec → ms)
        ├── ±500ms clamp
        ├── MAD (median absolute deviation) 계산
        └── reliable = MAD <= 50ms
        │
        ▼
        ├── reliable → UserPrefs.CalibrationOffsetMs = offset, Finish()
        └── !reliable
              ├── retry < 3 → "다시 해보세요" + 재실행
              └── retry == 3 → 0ms 저장 + Finish() (포기)
```

**왜 mean/stddev가 아니라 median/MAD인가**: 사용자 탭은 들쑥날쑥하다. 한 번 늦게 누른 outlier가 mean을 크게 왜곡한다. median은 outlier에 강하고, MAD는 분산이 아니라 편차의 중앙값이라 신뢰도 판정에 적합하다.

## 의존성 주입 패턴

런타임 코드의 모든 컴포넌트는 `[SerializeField]`로 의존성을 받는다. 이 SerializeField는 두 가지 방식으로 채워진다:

```
프로덕션
    SceneBuilder.Build()
        │
        ▼ AssetDatabase.LoadAssetAtPath / new GameObject + AddComponent
    GameObject 생성 + 컴포넌트 추가 + SerializeField 직접 대입
        │
        ▼
    Unity 에디터: 인스펙터에서 wired 상태로 보임
        │
        ▼ scene save
    GameplayScene.unity (.unity YAML, 커밋됨)


테스트 (EditMode)
    NUnit 테스트 [SetUp]
        │
        ▼ new GameObject + AddComponent
    빈 컴포넌트 인스턴스
        │
        ▼ SetDependenciesForTest(...) 호출
    SerializeField 자리에 mock / 실제 의존성 주입
        │
        ▼ TickForTest() 또는 InvokeXForTest(...)
    Update / private 메서드 결정적 호출
```

**왜 SceneBuilder가 단일 진실 소스인가** (SP7 결정):

W6 동안 두 차례 회귀가 발생했다.
- SP4: "모든 레인이 같은 피치로 재생" — `AudioSamplePool.pitchSamples` 와이어링이 `W6SamplesWireup.cs`라는 두 번째 메뉴 항목에서 처리됐는데, 이를 잊고 첫 번째 메뉴(`Build W4 Scene`)만 실행했다.
- SP6: 같은 패턴으로 196줄 와이어링 복원 디프 발생.

SP7에서 `W6SamplesWireup.cs`를 통째로 삭제하고 모든 와이어링을 `SceneBuilder.cs`로 흡수했다. 이제 메뉴 항목 1개만 실행하면 된다.

**규칙**: 새 컴포넌트가 씬 와이어링이 필요하면 `SceneBuilder.cs`에 `BuildX` 메서드를 추가하고 `Build`에서 호출되도록 한다. **새 `[MenuItem]` 헬퍼 툴을 만들지 않는다.**

## 에셋 임포트 후처리기

`.meta` 파일이 재생성될 수 있는 fresh checkout 상황에서도 임포트 설정이 살아남도록 후처리기로 강제한다.

```
파일 임포트 이벤트
        │
        ▼
AssetPostprocessor.OnPreprocess[Texture/Audio] 호출
        │
        ├── BackgroundImporterPostprocessor:
        │   - background_gameplay/yellow/start.png
        │   - 텍스처 → Sprite, ASTC 4x4, mipmap off
        │
        └── PianoSampleImportPostprocessor:
            - Assets/Audio/piano/*.wav
            - mono, Vorbis 60%, 48kHz, DecompressOnLoad
```

**규칙**: 위 대상 파일들의 임포트 설정을 인스펙터에서 손으로 바꾸지 말 것. 다음 임포트 때 덮어써진다.

## Android 권한 주입

`AndroidManifest.xml`을 직접 작성하지 않는다. Unity가 자동 생성하는 매니페스트에 후크로 권한을 추가한다.

```
ApkBuilder.Build()
        │
        ▼
BuildPipeline.BuildPlayer (Unity → gradle 프로젝트 export)
        │
        ▼
gradle 프로젝트 생성 후, 호출 콜백
        │
        ▼
AddVibratePermission.OnPostGenerateGradleAndroidProject(projectPath)
        │
        ├── projectPath/src/main/AndroidManifest.xml 로드
        ├── <uses-permission android:name="android.permission.VIBRATE"/> 추가
        └── 저장
        │
        ▼
gradle build → APK
```

**왜 `Assets/Plugins/Android/AndroidManifest.xml` 오버라이드가 아닌가**: 그 방식은 Unity의 자동 생성 launcher activity 블록을 통째로 대체한다. 새로운 Unity 버전에서 launcher 블록 형식이 바뀌면 매니페스트가 깨진다. 후크 방식은 Unity가 만든 매니페스트를 수정만 한다.
