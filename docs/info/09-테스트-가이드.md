# 09. 테스트 가이드

> 이 문서는 **개발자**가 새 EditMode 테스트를 추가하거나 기존 테스트를 수정할 수 있도록 정리했다. KeyFlow는 PlayMode 테스트가 없고 EditMode만 사용한다.

## 한눈에 보기

- **프레임워크**: NUnit (Unity Test Framework 1.6.0)
- **어셈블리**: `KeyFlow.Tests.EditMode` (`UNITY_INCLUDE_TESTS` 게이팅)
- **위치**: `Assets/Tests/EditMode/*.cs`
- **테스트 수**: 31개 클래스 (각 클래스 안에 여러 `[Test]`)
- **PlayMode**: 미존재 (지금까지 필요 없었음)
- **Python pytest**: `tools/midi_to_kfchart/tests/`에 별도 (10개 테스트)

## 어셈블리 정의

### `Assets/Tests/EditMode/KeyFlow.Tests.EditMode.asmdef`

```json
{
    "name": "KeyFlow.Tests.EditMode",
    "rootNamespace": "KeyFlow.Tests",
    "references": [
        "KeyFlow.Runtime",
        "KeyFlow.Editor",
        "UnityEngine.TestRunner",
        "UnityEditor.TestRunner",
        "Unity.Nuget.Newtonsoft.Json"
    ],
    "includePlatforms": ["Editor"],
    "precompiledReferences": ["nunit.framework.dll"],
    "autoReferenced": false,
    "defineConstraints": ["UNITY_INCLUDE_TESTS"]
}
```

요점:
- `includePlatforms: Editor` — 에디터에서만 컴파일, 게임 빌드에 미포함
- `defineConstraints: UNITY_INCLUDE_TESTS` — Unity가 테스트 컨텍스트에서만 정의하는 매크로
- `KeyFlow.Editor` 참조 — `SceneBuilder` 내부 동작 테스트 가능 (예: `CalibrationClickBuilderTests`)
- NUnit precompiled reference — Unity Test Framework가 NUnit을 같이 들여옴

## Internal 멤버 접근

### `[InternalsVisibleTo]`

`Assets/Scripts/Gameplay/JudgmentSystem.cs:6`에 선언:

```csharp
[assembly: System.Runtime.CompilerServices.InternalsVisibleTo("KeyFlow.Tests.EditMode")]
```

이 줄 하나로 **`KeyFlow.Runtime` 어셈블리 전체의 internal 멤버**가 테스트 어셈블리에 노출된다. `JudgmentSystem.cs`에 있는 건 임의 — 같은 어셈블리 내 어디든 한 번만 선언하면 됨.

### `internal` 메서드 활용

테스트 노출용 메서드는 `public` 대신 `internal`로 선언:

```csharp
// Production code
public class JudgmentSystem : MonoBehaviour
{
    private void HandleTap(int tapTimeMs, int tapLane) { /* ... */ }

    internal void InvokeHandleTapForTest(int tapTimeMs, int tapLane)
        => HandleTap(tapTimeMs, tapLane);
}
```

`public`이면 게임 코드 어디서나 호출 가능 → 의도와 어긋남. `internal`은 같은 어셈블리만 보고, `[InternalsVisibleTo]`로 테스트만 추가 허용.

## 테스트 훅 패턴

런타임 컴포넌트는 `#if UNITY_EDITOR || UNITY_INCLUDE_TESTS` 블록 안에 테스트 seam을 둔다. 두 가지 패턴이 반복된다.

### 패턴 1: `SetDependenciesForTest`

평소 `SerializeField`로 SceneBuilder가 주입하는 의존성을, 테스트가 직접 주입.

```csharp
public class HoldTracker : MonoBehaviour
{
    [SerializeField] private TapInputHandler tapInput;
    [SerializeField] private AudioSyncManager audioSync;
    [SerializeField] private JudgmentSystem judgmentSystem;
    [SerializeField] private AudioSamplePool audioPool;
    [SerializeField] private LaneGlowController laneGlow;

#if UNITY_EDITOR || UNITY_INCLUDE_TESTS
    internal void SetDependenciesForTest(
        AudioSyncManager audioSync,
        AudioSamplePool audioPool,
        LaneGlowController laneGlow = null,
        TapInputHandler tapInput = null,
        JudgmentSystem judgmentSystem = null)
    {
        this.audioSync = audioSync;
        this.audioPool = audioPool;
        this.laneGlow = laneGlow;
        this.tapInput = tapInput;
        this.judgmentSystem = judgmentSystem;
    }
#endif
}
```

#### 테스트에서

```csharp
[SetUp]
public void Setup() {
    var go = new GameObject("tracker");
    tracker = go.AddComponent<HoldTracker>();

    // Mock dependencies
    var sync = new GameObject("sync").AddComponent<AudioSyncManager>();
    var pool = new GameObject("pool").AddComponent<AudioSamplePool>();
    pool.InitializeForTest(channels: 4);

    tracker.SetDependenciesForTest(sync, pool);  // ← 여기
}
```

### 패턴 2: `TickForTest` / `InvokeXForTest`

`Update` 또는 private 메서드를 결정적으로 호출.

```csharp
public class HoldTracker : MonoBehaviour
{
    private void Update() { /* ... */ }

#if UNITY_EDITOR || UNITY_INCLUDE_TESTS
    internal void TickForTest() => Update();
#endif
}
```

#### 테스트에서

```csharp
[Test]
public void Hold_RetriggersAfter250ms() {
    tracker.OnHoldStartTapAccepted(note, tapTimeMs: 0);

    clock.DspTime = 0.25;  // 250ms 경과
    tracker.TickForTest();

    Assert.AreEqual(1, mockPool.PlayForPitchCallCount);
}
```

EditMode 테스트는 Unity 게임 루프가 안 돌아가므로 `Update`가 자동 호출되지 않는다. 명시적으로 호출.

### 패턴 3: 테스트 전용 프로퍼티

테스트가 검사할 필요가 있는 internal state를 노출.

```csharp
#if UNITY_EDITOR || UNITY_INCLUDE_TESTS
    internal int HoldAudioCountForTest => holdAudio.Count;
    internal bool PlayingForTest => playing;
#endif
```

## `ITimeSource` seam (가장 중요한 패턴)

`AudioSyncManager`의 시계는 인터페이스로 추상화돼 있다.

```csharp
// Production
public interface ITimeSource { double DspTime { get; } }

internal class RealTimeSource : ITimeSource {
    public double DspTime => AudioSettings.dspTime;
}

public class AudioSyncManager : MonoBehaviour {
    public ITimeSource TimeSource { get; set; } = new RealTimeSource();
    // ...
}
```

테스트는 수동 클럭을 주입:

```csharp
private class ManualClock : ITimeSource
{
    public double DspTime { get; set; }
}

[SetUp]
public void Setup() {
    sync = go.AddComponent<AudioSyncManager>();
    clock = new ManualClock { DspTime = 100.0 };
    sync.TimeSource = clock;
}

[Test]
public void SongTimeMs_IsFrozenWhilePaused() {
    sync.StartSilentSong();
    clock.DspTime = 101.0;
    int t0 = sync.SongTimeMs;

    sync.Pause();
    clock.DspTime = 105.0;  // 4초 흐름
    Assert.AreEqual(t0, sync.SongTimeMs);  // 동결 확인
}
```

이 seam은 Unity 오디오 스레드 없이 dspTime 진행을 시뮬레이션하게 해준다. 모든 시간 의존 테스트(`AudioSyncPauseTests`, `GameTimeTests`, `HoldAudioRetriggerTests`, `GameplayControllerCountdownTests`)가 이 패턴을 사용.

## 실행 방법

### Unity 에디터에서

1. Window → General → Test Runner
2. EditMode 탭 선택
3. "Run All" 또는 개별 테스트 선택 → "Run Selected"

녹색 = 통과, 빨강 = 실패. 실패 시 해당 행 클릭 → Stack trace + Message 확인.

### Batch-mode CLI

```bash
"<UnityEditor>" -batchmode -nographics -projectPath . \
  -runTests -testPlatform EditMode \
  -testResults Builds/test-results.xml
```

**주의**: `-quit`를 빼야 한다. `-quit`가 있으면 Unity가 테스트 러너를 시작하기 전에 종료한다.

테스트 결과는 NUnit XML 형식으로 `Builds/test-results.xml`에 저장. CI에서 파싱 가능.

### 단일 테스트 클래스만

```bash
"<UnityEditor>" -batchmode -nographics -projectPath . \
  -runTests -testPlatform EditMode \
  -testFilter "KeyFlow.Tests.EditMode.JudgmentEvaluatorTests" \
  -testResults Builds/test-results.xml
```

`-testFilter`는 정규식 매칭 (NUnit 카테고리/네임스페이스/클래스/메서드 이름).

## 새 테스트 추가 절차

1. `Assets/Tests/EditMode/<Subject>Tests.cs` 파일 생성
2. 네임스페이스: `KeyFlow.Tests.EditMode`
3. NUnit 어트리뷰트:
   ```csharp
   using NUnit.Framework;

   namespace KeyFlow.Tests.EditMode
   {
       public class MySystemTests
       {
           [SetUp]      public void Setup() { /* ... */ }
           [TearDown]   public void Teardown() { /* ... */ }

           [Test]
           public void Behaves_Correctly_When_Conditions() {
               // arrange
               // act
               // assert
               Assert.AreEqual(expected, actual);
           }
       }
   }
   ```
4. SerializeField 의존성이 필요하면 → 대상 클래스에 `SetDependenciesForTest` 추가
5. `Update` 호출이 필요하면 → 대상 클래스에 `TickForTest` 추가
6. dspTime이 필요하면 → `ManualClock : ITimeSource` 패턴
7. Unity 에디터 Test Runner에서 실행 → 통과 확인

### 명명 규칙

- 클래스: `<Subject>Tests` (예: `JudgmentSystemTests`)
- 메서드: `<Action>_<Conditions>_<ExpectedOutcome>` (예: `OnStartTapAccepted_TransitionsToHolding`)

### TearDown에서 GameObject 정리

```csharp
[TearDown]
public void Teardown() { Object.DestroyImmediate(go); }
```

EditMode 테스트는 같은 에디터 세션에서 연속 실행되므로 GameObject가 누적되면 다음 테스트에 영향. 반드시 `DestroyImmediate`(즉시 destroy, EditMode에서 필수).

## 왜 PlayMode 테스트가 없나

PlayMode는 실제 Unity 게임 루프에서 동작 — 카메라 렌더, 오디오 스레드, 프레임 스케줄링 모두 활성화. 그래서:
- 실행 시간이 EditMode보다 훨씬 김
- 결정성이 떨어짐(프레임 타이밍 의존)
- CI 환경 설정이 복잡 (그래픽 드라이버 등)

KeyFlow는 모든 핵심 로직이 `ITimeSource` seam으로 시간 의존성을 분리해뒀기 때문에 EditMode + 수동 클럭으로 충분히 검증 가능. 실제 오디오 동기 / 입력 응답성은 **수동 기기 테스트**(Galaxy S22)로 검증.

만약 PlayMode가 필요해지면 `Assets/Tests/PlayMode/` 폴더와 별도 `.asmdef` 추가 (`includePlatforms: Editor + Standalone + Android`, `referencedAssemblies`에 `UnityEngine.TestRunner` 등).

## 디버깅 팁

### 테스트 실패 시

1. Test Runner에서 실패 행 클릭 → Stack trace 확인
2. 라인 번호로 점프 (또는 IDE에서 break point)
3. `Debug.Log("[Test] ...")` 추가 → Test Runner의 Console 탭에서 확인

### 한 테스트가 다른 테스트에 영향 줄 때

`[SetUp]`/`[TearDown]`이 정상 동작하는지 확인. 정적 상태(`SongCatalog.loaded`, `SongSession.CurrentSongId`, `UserPrefs`의 PlayerPrefs)는 Setup에서 초기화 필요.

```csharp
[SetUp]
public void Setup() {
    SongSession.Reset();
    PlayerPrefs.DeleteAll();
    SongCatalog.SetForTesting(new SongEntry[0]);
}
```

### 시간 진행 결정성

`Time.time` / `Time.deltaTime`은 EditMode에서 **0이거나 예측 불가**. 시간 의존 코드를 테스트하려면:
- `AudioSyncManager.TimeSource = new ManualClock()`
- 또는 컴포넌트의 `SetTimeForTest(double t)` 식 seam 추가
