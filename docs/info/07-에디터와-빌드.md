## 1. 에디터 메뉴 6개

Unity 에디터를 열면 상단 메뉴바에 **`KeyFlow/`** 항목이 추가된다. `Assets/Editor/` 하위 클래스들이 `[MenuItem("KeyFlow/...")]` 어트리뷰트로 등록한 6개 메뉴:

| 메뉴 | 클래스 / 메서드 | 역할 |
|---|---|---|
| `KeyFlow/Build W4 Scene` | `SceneBuilder.Build` | `GameplayScene.unity` 통째 재생성 |
| `KeyFlow/Build APK` | `ApkBuilder.Build` | Release APK 빌드 |
| `KeyFlow/Build APK (Profile)` | `ApkBuilder.BuildProfile` | Profiler + Debug APK 빌드 |
| `KeyFlow/Build Calibration Click` | `CalibrationClickBuilder.Build` | `calibration_click.wav` 결정적 생성 |
| `KeyFlow/Build Feedback Assets` | `FeedbackPrefabBuilder.Build` | hit/miss 파티클 prefab + `FeedbackPresets.asset` 생성 |
| `KeyFlow/Apply Android Icon (SP9)` | `SP9IconSetter.Apply` | 모든 Android 아이콘 슬롯에 `Assets/Textures/icon.png` 채움 |

추가로 `[MenuItem]`은 없지만 자동으로 동작하는 컴포넌트들:

| 클래스 | 역할 |
|---|---|
| `BackgroundImporterPostprocessor` | 배경 PNG의 임포트 설정 강제 (ASTC 4x4, mipmap off) |
| `PianoSampleImportPostprocessor` | 피아노 WAV의 임포트 설정 강제 (mono, Vorbis 60%, 48kHz, DecompressOnLoad) |
| `AddVibratePermission` | gradle 프로젝트 생성 시 VIBRATE 권한 주입 |

## 2. SceneBuilder — 단일 진실 소스

### 핵심 원칙

`Assets/Editor/SceneBuilder.cs`가 `GameplayScene.unity`를 **처음부터 통째로 재생성**한다. 즉:

- 인스펙터에서 손으로 만든 GameObject는 다음 빌드 시 **사라진다**.
- 모든 컴포넌트 와이어링은 코드에 있다.
- 씬 변경 = 코드 변경 → 메뉴 재실행 → 디프 커밋.

### `Build()` 흐름 (단순화)

```csharp
[MenuItem("KeyFlow/Build W4 Scene")]
public static void Build() {
    EnsureFolder("Assets/Scenes");
    EnsureFolder("Assets/Prefabs");
    // ... 폴더 보장

    // 1. 사전 자산 로드 (없으면 abort + 안내 메시지)
    var pianoClip = AssetDatabase.LoadAssetAtPath<AudioClip>("Assets/Audio/piano_c4.wav");
    var clickClip = AssetDatabase.LoadAssetAtPath<AudioClip>("Assets/Audio/calibration_click.wav");
    var blueBg    = AssetDatabase.LoadAssetAtPath<Sprite>("Assets/Sprites/background_gameplay.png");
    // ...

    // 2. 새 씬 시작
    var scene = EditorSceneManager.NewScene(NewSceneSetup.EmptyScene, NewSceneMode.Single);

    // 3. GameObject + 컴포넌트 + 와이어링
    var camera = BuildMainCamera();
    CreateEventSystem();
    var backgroundSwitcher = BuildBackgroundCanvas(blueBg, yellowBg, camera);
    var gameplayRoot = new GameObject("GameplayRoot");

    BuildLaneDividers(whiteSprite, gameplayRoot.transform);
    BuildJudgmentLine(whiteSprite, gameplayRoot.transform);
    BuildManagers(/*...*/, out audioSync, out samplePool, out tapInput,
                            out judgmentSystem, out spawner, out holdTracker);
    BuildFeedbackPipeline(camera, judgmentSystem, gameplayRoot.transform);
    BuildHUD(audioSync, tapInput, samplePool, judgmentSystem, /*...*/);
    BuildCountdownOverlay(camera, clickClip, /*...*/);
    BuildCalibrationOverlay(whiteSprite, clickClip, audioSync);
    BuildResultsCanvas(whiteSprite);
    BuildGameplayController(/*...*/);
    BuildMainCanvas(whiteSprite);
    BuildPauseCanvas(whiteSprite, audioSync);
    BuildSettingsCanvas(whiteSprite, calibration);
    BuildStartCanvas(startBg, bgmClip, out startScreen);

    // 4. ScreenManager 와이어
    var screenMgr = new GameObject("ScreenManager").AddComponent<ScreenManager>();
    SetField(screenMgr, "startRoot", startCanvas);
    // ... 모든 SerializeField 채움

    // 5. 저장
    EditorSceneManager.SaveScene(scene, "Assets/Scenes/GameplayScene.unity");
}
```

### `SetField` 헬퍼

`SerializeField`는 private/protected라 외부에서 직접 대입 불가. SceneBuilder는 리플렉션으로 우회:

```csharp
private static void SetField(object target, string fieldName, object value) {
    var field = target.GetType().GetField(
        fieldName,
        BindingFlags.NonPublic | BindingFlags.Instance);
    field.SetValue(target, value);
}
```

이 패턴은 코드가 SerializeField 명을 알아야 한다는 단점이 있지만, 인스펙터-기반 와이어링의 위험(누락, 잘못된 참조)을 컴파일 타임으로 옮긴다.

### SP7 통합 결정

원래는 메뉴 두 개(`KeyFlow/Build W4 Scene` + `KeyFlow/W6 Samples Wireup`)를 순서대로 실행해야 했다. 두 번째를 잊는 회귀가 W6 동안 두 번 발생:

| SP | 증상 | 원인 |
|---|---|---|
| SP4 | "모든 레인이 같은 피치로 재생" | `AudioSamplePool.pitchSamples` 미와이어 |
| SP6 | 196줄 와이어링 복원 디프 발생 | 같은 패턴 |

SP7에서 두 번째 메뉴(`W6SamplesWireup.cs`)를 통째 삭제하고 `BuildManagers`로 흡수했다. 이제 메뉴 한 개만 실행하면 된다.

**규칙**: 새 컴포넌트가 씬 와이어링이 필요하면:
1. `SceneBuilder.cs`에 `BuildX(...)` 메서드 추가
2. `Build()`에서 호출
3. 출력값은 `out var`로 반환해 후속 와이어링에서 사용

새 `[MenuItem]` 헬퍼 툴을 만들지 말 것 — 동일 트랩이 다시 생긴다.

## 3. 최초 세팅 순서

새 워크트리를 받으면 이 순서로 메뉴 실행:

```
1. KeyFlow/Build Calibration Click
   → Assets/Audio/calibration_click.wav 생성

2. KeyFlow/Build Feedback Assets
   → Assets/Prefabs/Feedback/{hit,miss}.prefab + ScriptableObjects/FeedbackPresets.asset 생성

3. KeyFlow/Build W4 Scene
   → Assets/Scenes/GameplayScene.unity 생성

4. KeyFlow/Build APK
   → Builds/keyflow-w<X>-sp<Y>.apk 생성
```

1·2를 빠뜨리면 3에서 `Missing Assets/Audio/calibration_click.wav. Run KeyFlow/Build Calibration Click first. Aborting.` 식의 에러 + abort.

### 임포트 설정 강제 재실행

워크트리 fresh checkout 후 piano WAV 임포트 설정이 깨졌다면:

```bash
"<UnityEditor>" -batchmode -nographics -projectPath . \
  -executeMethod KeyFlow.Editor.PianoSampleImportPostprocessor.ForceReimportPianoSamples -quit
```

`AssetDatabase.ImportAsset(path, ImportAssetOptions.ForceUpdate)`을 17개 WAV에 대해 호출 → `OnPreprocessAudio` 재실행 → 설정 메타에 직렬화.

## 4. 에셋 임포트 후처리기

### `BackgroundImporterPostprocessor` (`Assets/Editor/BackgroundImporterPostprocessor.cs`)

대상: `Assets/Sprites/background_{gameplay,yellow,start}.png`.

```csharp
private void OnPreprocessTexture() {
    if (Array.IndexOf(TargetPaths, assetPath) < 0) return;

    var importer = (TextureImporter)assetImporter;
    importer.textureType = TextureImporterType.Sprite;
    importer.mipmapEnabled = false;
    importer.isReadable = false;

    var androidSettings = importer.GetPlatformTextureSettings("Android");
    androidSettings.overridden = true;
    androidSettings.format = TextureImporterFormat.ASTC_4x4;
    androidSettings.compressionQuality = 50;
    importer.SetPlatformTextureSettings(androidSettings);
}
```

ASTC 4x4 = 모바일 GPU에서 빠른 디코드 + 적당한 압축률.

### `PianoSampleImportPostprocessor` (`Assets/Editor/PianoSampleImportPostprocessor.cs`)

대상: `Assets/Audio/piano/*.wav` 17개 모두.

```csharp
private void OnPreprocessAudio() {
    if (!assetPath.StartsWith("Assets/Audio/piano/")) return;

    var importer = (AudioImporter)assetImporter;
    importer.forceToMono = true;

    var defaults = importer.defaultSampleSettings;
    defaults.loadType = AudioClipLoadType.DecompressOnLoad;
    defaults.compressionFormat = AudioCompressionFormat.Vorbis;
    defaults.quality = 0.60f;
    defaults.sampleRateSetting = AudioSampleRateSetting.OverrideSampleRate;
    defaults.sampleRateOverride = 48000;
    defaults.preloadAudioData = true;
    importer.defaultSampleSettings = defaults;
}
```

`DecompressOnLoad`로 게임 시작 시점에 메모리에 풀어둠 → 재생 시 디스크 IO 0.

`preloadAudioData = true` — Unity 6 변경점: 기본 설정이 아니라 `SampleSettings` 안으로 들어감.

### 왜 인스펙터 직접 설정이 아닌가

`.meta` 파일이 재생성될 수 있는 시나리오(다른 워크트리에서 동일 파일 재임포트, 또는 `.meta` 누락 후 재생성)에서 인스펙터 설정은 default로 돌아간다. 후처리기가 매 임포트마다 강제하면 .meta가 어떻게 되든 결과가 같다.

## 5. Android 권한 주입 (gradle 후크)

### `AddVibratePermission` (`Assets/Editor/AddVibratePermission.cs`)

`IPostGenerateGradleAndroidProject` 인터페이스 구현. Unity가 gradle 프로젝트를 export한 직후 콜백.

```csharp
public class AddVibratePermission : IPostGenerateGradleAndroidProject
{
    public int callbackOrder => 1;

    public void OnPostGenerateGradleAndroidProject(string projectPath) {
        string manifestPath = Path.Combine(projectPath, "src", "main", "AndroidManifest.xml");
        var doc = new XmlDocument();
        doc.Load(manifestPath);

        // 이미 있으면 skip
        foreach (XmlNode child in doc.DocumentElement.ChildNodes) {
            if (child.Name == "uses-permission"
                && child.Attributes?["android:name"]?.Value == "android.permission.VIBRATE")
                return;
        }

        // <uses-permission android:name="android.permission.VIBRATE" /> 추가
        var permission = doc.CreateElement("uses-permission");
        var attr = doc.CreateAttribute("android", "name", "http://schemas.android.com/apk/res/android");
        attr.Value = "android.permission.VIBRATE";
        permission.Attributes.Append(attr);
        doc.DocumentElement.PrependChild(permission);

        doc.Save(manifestPath);
    }
}
```

### 왜 `Assets/Plugins/Android/AndroidManifest.xml` 오버라이드가 아닌가

오버라이드 방식의 단점:
- Unity가 자동 생성하는 `<application>` 블록 + launcher activity 정의를 **통째로 대체**해야 함
- Unity 버전 업그레이드 시 launcher 정의 형식이 바뀌면 매니페스트 깨짐
- 새 권한 추가 시 매번 풀 매니페스트 편집

후크 방식의 장점:
- Unity가 만든 매니페스트를 **수정만** 함
- 멱등(이미 있으면 skip)
- 새 권한 = 새 후크 클래스 추가 (서로 독립)

**새 Android 권한 추가 시**: `AddVibratePermission` 패턴을 그대로 복사해서 새 클래스 만들기. `callbackOrder`만 다른 값으로.

## 6. APK 빌더

### `ApkBuilder` (`Assets/Editor/ApkBuilder.cs`)

```csharp
[MenuItem("KeyFlow/Build APK")]
public static void Build() {
    string apk = "Builds/keyflow-w6-sp12.apk";  // ← 하드코딩

    var opts = new BuildPlayerOptions {
        scenes = new[] { "Assets/Scenes/GameplayScene.unity" },
        locationPathName = apk,
        target = BuildTarget.Android,
        options = BuildOptions.None
    };

    var report = BuildPipeline.BuildPlayer(opts);
    if (report.summary.result != BuildResult.Succeeded)
        throw new Exception($"APK build failed: {report.summary.result}");
}

[MenuItem("KeyFlow/Build APK (Profile)")]
public static void BuildProfile() {
    // 같은 패턴, 단:
    // - 출력: keyflow-w6-sp12-profile.apk
    // - options: Development | ConnectWithProfiler | AllowDebugging
}
```

### 파일명 컨벤션

`keyflow-w<X>-sp<Y>.apk` 형식. SP 마일스톤마다 SP12 → SP13 식으로 **하드코딩 직접 수정**해야 함. SP7 reports에서 명시:
> "SP4 carry-over — APK filename bump (`keyflow-w6-sp2.apk`). Cosmetic for release tracking."

자동화하지 않은 이유: 빌드 산출물이 역사 추적 가능하도록 명시적으로 SP를 박아두고 싶었음.

### Profile 빌드 vs Release 빌드

| 옵션 | Release | Profile |
|---|---|---|
| `Development` 플래그 | x | o |
| `ConnectWithProfiler` | x | o |
| `AllowDebugging` | x | o |
| Symbol stripping | 기본값 | 비활성 |
| 용도 | 사용자 배포 | Unity Profiler로 프레임/메모리 분석 |

## 7. Batch-mode CLI

CI / 헤드리스 / 자동화 환경에서 Unity를 GUI 없이 돌린다.

### 공통 옵션

```bash
"<UnityEditor>" \
  -batchmode \           # GUI 안 띄움
  -nographics \          # 렌더 컨텍스트 안 만듬 (서버용)
  -projectPath . \       # 프로젝트 루트 지정
  -executeMethod KeyFlow.Editor.<Class>.<Method> \
  -quit                  # 메서드 끝나면 종료
```

### 4가지 워크플로

```bash
# 1. 씬 재생성
"<UnityEditor>" -batchmode -nographics -projectPath . \
  -executeMethod KeyFlow.Editor.SceneBuilder.Build -quit

# 2. Release APK
"<UnityEditor>" -batchmode -nographics -projectPath . \
  -executeMethod KeyFlow.Editor.ApkBuilder.Build -quit

# 3. 피아노 WAV 강제 재임포트
"<UnityEditor>" -batchmode -nographics -projectPath . \
  -executeMethod KeyFlow.Editor.PianoSampleImportPostprocessor.ForceReimportPianoSamples -quit

# 4. EditMode 테스트 → Builds/test-results.xml
"<UnityEditor>" -batchmode -nographics -projectPath . \
  -runTests -testPlatform EditMode -testResults Builds/test-results.xml
```

### 치명적 함정: `-runTests` + `-quit`

테스트 실행에는 **`-quit`를 빼야 한다**. `-quit`가 있으면 Unity가 테스트 러너를 시작하기 전에 종료한다. 다른 모든 `-executeMethod`는 `-quit` 필수.

### macOS / Windows 실행 파일 경로

```bash
# macOS
/Applications/Unity/Hub/Editor/6000.3.13f1/Unity.app/Contents/MacOS/Unity

# Windows
"C:\Program Files\Unity\Hub\Editor\6000.3.13f1\Editor\Unity.exe"
```

쉘 스크립트로 추상화하는 게 일반적이지만 현재 저장소엔 그런 스크립트가 없음 — 수동 실행.

## 8. 기기 설치 + 디버깅

### `adb install -r`

```bash
adb install -r Builds/keyflow-w6-sp12.apk
```

`-r`은 reinstall (이전 버전 위에 덮어씀, 데이터 보존). 새 버전 같은 패키지명이면 자동 덮어씀.

### USB 디버깅 활성화

기기에서:
1. 설정 → 휴대전화 정보 → 빌드 번호 7번 탭 (개발자 옵션 활성화)
2. 설정 → 개발자 옵션 → USB 디버깅 ON

기기 연결 후 `adb devices`로 인식 확인.

### 로그 보기

```bash
adb logcat -s Unity   # Unity 로그만 필터링
adb logcat | grep KeyFlow   # KeyFlow 태그 필터
```

`Debug.Log("[KeyFlow] ...")` 형식으로 코드에 박아두면 grep 필터링이 쉬움.

### 프로파일링 연결

Profile 빌드 설치 후 Unity 에디터에서 Window → Analysis → Profiler → Connected Player 선택. Frame 단위 CPU/GPU/메모리 그래프 확인 가능.
