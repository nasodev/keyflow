# 08. 채보 파이프라인

> 이 문서는 **개발자 또는 채보 작업자**가 새 곡을 추가하거나 기존 채보를 튜닝할 수 있도록 정리했다. Python 8단계 파이프라인의 각 단계와 `.kfchart` JSON 형식을 다룬다.

## 파이프라인 8단계

각 단계는 list[dict] 입력 → list[dict] 출력의 순수 함수. 부수효과 없음, 같은 입력 → 같은 출력.

### 1. `parser.parse_midi(path) → list[RawNote]`

`mido`로 MIDI 파일을 시간 순 RawNote 리스트로 변환.

```python
RawNote = {"t_ms": int, "pitch": int, "dur_ms": int}
```

`note_on` 시작 시 active 딕셔너리에 기록 → `note_off` 또는 `velocity=0` `note_on` 시 종료. duration 계산 후 list에 추가.

### 2. `chord_reducer.collapse(raws) → list[RawNote]`

같은 시각의 여러 음(코드)을 **가장 높은 피치 한 개**로 축약.

```python
def collapse(raws):
    by_time = {}
    for n in raws:
        existing = by_time.get(n["t_ms"])
        if existing is None or n["pitch"] > existing["pitch"]:
            by_time[n["t_ms"]] = n
    return [by_time[t] for t in sorted(by_time)]
```

KeyFlow는 모노포닉(한 번에 한 음). 코드는 멜로디 라인이 아니라 반주이므로 가장 높은 음(=멜로디)을 남긴다.

### 3. `hold_detector.classify(raws) → list[Note]`

duration이 임계 초과면 HOLD, 아니면 TAP.

```python
HOLD_THRESHOLD_MS = 500
HOLD_CAP_MS = 4000

# 500ms 이상 → HOLD, 단 4000ms로 cap
```

| 입력 dur | type | 출력 dur |
|---|---|---|
| < 500ms | TAP | 0 |
| 500–4000ms | HOLD | 원본 |
| > 4000ms | HOLD | **4000으로 cap** |

cap이 있는 이유: 너무 긴 HOLD는 게임플레이 화면을 다 차지하고 사용자가 손을 너무 오래 누르고 있어야 함.

### 4. `density.thin(notes, target_nps, duration_ms) → list[Note]`

**NPS(Notes Per Second) 목표치에 맞춰 노트 솎아냄.** 결정적, 무작위 없음.

```python
def thin(notes, target_nps, duration_ms):
    duration_sec = duration_ms / 1000.0
    current_nps = len(notes) / duration_sec
    allowed_nps = target_nps * 1.1  # 10% 마진

    if current_nps <= allowed_nps:
        return list(notes)

    keep_ratio = allowed_nps / current_nps
    step = round(1 / (1 - keep_ratio))
    if step < 2:
        return []  # ratio 너무 극단적

    return [n for i, n in enumerate(notes) if (i + 1) % step != 0]
```

알고리즘: keep_ratio가 0.8이면 step = 5, 즉 5번째 노트마다 제거. 결정적이고 시간 분포가 균일하다.

이 단계가 **튜닝 루프의 핵심**이다. `batch.yaml`의 `target_nps`를 조정해서 곡의 난이도를 바꾼다 — 채보 JSON을 직접 편집하지 않는다.

### 5. `pitch_clamp.clamp_pitches(notes) → list[Note]`

MIDI 36–83 범위로 옥타브 단위 transpose.

```python
PITCH_MIN = 36
PITCH_MAX = 83

def clamp_pitches(notes):
    for n in notes:
        p = n["pitch"]
        while p < PITCH_MIN: p += 12   # 한 옥타브 위로
        while p > PITCH_MAX: p -= 12   # 한 옥타브 아래로
```

옥타브(12 반음)로 이동하므로 음악적 관계는 유지된다. 36–83 범위는 `AudioSamplePool`의 17개 샘플(C2–C6)이 커버하는 영역.

### 6. `lane_assigner.assign(notes) → list[Note]`

피치 분포를 4분위로 나눠 레인 0–3에 매핑 + 같은 레인 3연속 방지.

```python
def assign(notes):
    pitches = [n["pitch"] for n in notes]
    pmin, pmax = min(pitches), max(pitches)

    # 4분위 매핑: 낮은 피치 → lane 0, 높은 피치 → lane 3
    for n in out:
        ratio = (n["pitch"] - pmin) / (pmax - pmin)
        n["lane"] = min(LANE_COUNT - 1, int(ratio * LANE_COUNT))

    # 3-consecutive relief: 같은 레인이 3번 연속이면 다음 레인으로
    for i in range(2, len(out)):
        if out[i]["lane"] == out[i-1]["lane"] == out[i-2]["lane"]:
            candidate = (out[i]["lane"] + 1) % LANE_COUNT
            out[i]["lane"] = candidate
```

**왜 4분위인가**: 피아노곡의 자연스러운 멜로디는 한 옥타브 안에서 움직인다. 그 범위를 4등분하면 비슷한 피치는 같은 레인, 점프는 다른 레인이 된다 — 손가락 이동이 곡 흐름과 일치한다.

**왜 3연속 relief인가**: 같은 레인을 4번 이상 연속 두드리면 단조롭고 손가락 피로 유발. 3연속 후엔 옆 레인으로 옮긴다.

### 7. `merge_adjacent_holds.merge(notes) → list[Note]`

같은 레인의 인접 HOLD를 하나로 병합.

```python
MERGE_GAP_MS = 500
MAX_DUR_MS = 4000

# 같은 레인 HOLD 두 개의 gap이 < 500ms이면 하나로 병합 (총 길이 4000ms cap)
```

**왜 필요한가**: 같은 레인 HOLD 두 개가 연속되면 사용자에겐 하나의 긴 노트로 보인다. 그러나 게임 입력 상태기는 두 번째 HOLD에 새 시작 탭을 기대 — 사용자는 누른 채로 있으므로 시작 탭이 안 와서 두 번째가 자동 미스. 이걸 방지하려고 병합.

이 단계는 SP8(hold-note-polish)에서 도입.

### 8. `emitter.write_kfchart(notes, meta, difficulty, out|merge_into)`

JSON 직렬화 + 파일 저장.

```python
{
    "songId": "beethoven_fur_elise",
    "title": "엘리제를 위하여",
    "composer": "Beethoven",
    "bpm": 72,
    "durationMs": 45000,
    "charts": {
        "NORMAL": {
            "totalNotes": 158,
            "notes": [
                {"t": 0, "lane": 2, "pitch": 64, "type": "TAP", "dur": 0},
                ...
            ]
        }
    }
}
```

저장 시 `_validate`가 다음을 검증(`emitter.py:35–52`):
- 노트 비어있지 않음
- t 오름차순
- lane ∈ [0, 3]
- pitch ∈ [36, 83]
- TAP은 dur=0, HOLD는 dur>0
- type ∈ {"TAP", "HOLD"}

### `--out` vs `--merge-into`

```python
if (out is None) == (merge_into is None):
    raise ValueError("exactly one of --out or --merge-into must be set")
```

| 옵션 | 동작 |
|---|---|
| `--out new.kfchart` | 새 파일 생성 (단일 난이도) |
| `--merge-into existing.kfchart` | 기존 파일에 난이도 추가/덮어씀 |

같은 곡의 EASY와 NORMAL을 한 파일에 합치려면 `--merge-into` 두 번 호출.

## 튜닝 루프

```
1. tools/midi_to_kfchart/batch_*.yaml 의 target_nps 조정
2. python midi_to_kfchart.py --batch batch_*.yaml
   → Assets/StreamingAssets/charts/<id>.kfchart 갱신
3. Unity 에디터에서 Play 또는 APK 빌드 → 기기 설치
4. 플레이해보고 너무 빡세면 target_nps 낮추기 / 너무 쉬우면 올리기
5. 1로 돌아감
```

### 절대 하지 말 것

- **`.kfchart` JSON 직접 편집**: 다음 파이프라인 실행 시 덮어써진다. 손작성 채보(엘리제)는 예외 — `tools/`에서 처리하지 않는다.
- **`PITCH_MIN/MAX` 변경**: `AudioSamplePool`의 17개 샘플 범위와 동기화 필요. 한쪽만 바꾸면 클램프 실패.
- **`HOLD_THRESHOLD_MS` / `HOLD_CAP_MS` 임의 변경**: 게임플레이 균형(`HoldStateMachine`의 250ms retrigger)에 영향.

## 새 곡 추가 절차

1. **MIDI 확보**: `tools/midi_to_kfchart/midi_sources/<song>.mid` 추가. PD 또는 라이선스 호환 출처만.
2. **출처 기록**: `tools/midi_to_kfchart/midi_sources/SOURCES.md`에 attribution 추가.
3. **batch YAML 항목 추가**: 새 `song_id`, `midi`, `bpm`, `duration_ms`, `difficulties` 작성.
4. **파이프라인 실행**: `python midi_to_kfchart.py --batch batch_*.yaml`.
5. **catalog 업데이트**: `Assets/StreamingAssets/catalog.kfmanifest`에 `{id, title, composer, thumbnail, difficulties, chartAvailable}` 항목 추가.
6. **썸네일 생성**: `tools/gen_thumbs.py`의 `TARGETS`에 새 곡 추가 + `python tools/gen_thumbs.py`.
7. **테스트**: Unity에서 `MainScreen` → 새 카드 표시 확인 → Play.

## 손작성 채보: 엘리제 (Für Elise)

`Assets/StreamingAssets/charts/beethoven_fur_elise.kfchart`는 파이프라인 출력이 아니라 **손으로 작성**한 JSON이다.

이유:
- 곡이 짧고(45초) 멜로디가 명확
- target_nps 자동 조정만으로는 도입부 "Mi-Re#-Mi-Re#" 같은 시그니처 패턴이 망가짐
- 손작성이 더 빠름

작곡 자체는 PD라 라이선스 문제 없음.

손작성 채보를 수정하려면 JSON 직접 편집(파이프라인 실행 안 됨). `emitter._validate`와 동일 규칙 따라야 함.
