## 1. 입력

### `TapInputHandler` (`Assets/Scripts/Gameplay/TapInputHandler.cs`)

Unity Input System으로 터치/마우스를 잡아 **레인 인덱스(0–3)** + **곡 시각(ms)**으로 변환한다.

#### 핵심 동작

```csharp
private void Update()
{
    // 일시정지 중이거나 오버레이 떠 있으면 입력 무시
    if (audioSync.IsPaused) return;
    if (ScreenManager.Instance.AnyOverlayVisible) return;

    int songTimeMs = audioSync.SongTimeMs;

    // 멀티터치 지원
    foreach (var touch in Touchscreen.current.touches) {
        if (touch.press.wasPressedThisFrame) {
            Vector2 pos = touch.position.ReadValue();
            if (IsOverUI(pos)) continue;  // UI 위 탭은 무시
            int lane = ScreenToLane(pos);
            FirePress(touch.touchId.ReadValue(), lane, songTimeMs);
        }
        else if (touch.press.wasReleasedThisFrame) {
            FireRelease(touch.touchId.ReadValue());
        }
    }
}
```

#### 발화 이벤트

| 이벤트 | 시그니처 | 누가 듣는가 |
|---|---|---|
| `OnTap` | `Action<int songTimeMs>` | (현재 미사용) |
| `OnLaneTap` | `Action<int songTimeMs, int lane>` | `JudgmentSystem.HandleTap` |
| `OnLaneRelease` | `Action<int lane>` | (홀드는 `HoldTracker`가 `IsLanePressed` 폴링) |

#### 탭 시 즉시 음 재생

`FirePress`는 이벤트 발화 **전에** `PlayTapSound`를 호출한다. 즉 판정과 무관하게 모든 탭은 소리가 난다 — 입력 응답성을 위함.

```csharp
private void PlayTapSound(int lane, int songTimeMs)
{
    // 500ms 윈도 내 가장 가까운 pending 노트의 피치를 가져옴
    int pitch = judgmentSystem.GetClosestPendingPitch(lane, songTimeMs, 500);

    // pending 노트가 없으면 레인별 기본 피치
    if (pitch < 0) pitch = LanePitches.Default(lane);

    samplePool.PlayForPitch(pitch);
}
```

**왜 이런 설계인가**: 탭과 판정이 비동기적으로 처리돼도(판정 코드가 3ms 걸려도) 음은 즉시 들린다. 사용자는 탭 시점의 피드백을 가장 중요하게 느낀다.

#### 멀티터치 처리

`touchToLane` 딕셔너리(touchId → lane)와 `pressedLanes` 해시셋을 구분 관리한다. 같은 레인에 두 손가락이 있을 때 한 개만 떼도 `pressedLanes`에는 남는다 — 홀드가 깨지지 않도록.

## 2. 노트 스폰 + 시각화

### `NoteSpawner` (`Assets/Scripts/Gameplay/NoteSpawner.cs`)

채보를 받아 **2초 앞서**(`previewMs = 2000`) `Note.prefab`을 인스턴스화한다.

```csharp
private void Update()
{
    if (!audioSync.IsPlaying || audioSync.IsPaused) return;
    if (spawnedCount >= chart.notes.Count) return;

    var next = chart.notes[spawnedCount];
    if (audioSync.SongTimeMs >= next.t - previewMs)
    {
        SpawnNote(next);
        spawnedCount++;
    }
}
```

스폰 시 `JudgmentSystem.RegisterPendingNote`로 등록 + `liveNotes`에 추가(retry 시 일괄 destroy 위해).

### `NoteController` (`Assets/Scripts/Gameplay/NoteController.cs`)

생성된 노트 1개의 **위치 보간 + 자동 미스 + 시각 효과**.

#### 위치 계산

```
spawnY ────────────► judgmentY
   ↓
progress = 1 - (hitTimeMs - songTimeMs) / previewMs

position.y = Lerp(spawnY, judgmentY, progress) + holdHalfExtent
```

`progress < 0`(스폰 시각 이전)이면 위치 갱신 안 함. `progress > 1`(판정선 통과 후)도 계속 떨어진다 — `LerpUnclamped` 사용.

#### HOLD 노트 시각 처리

HOLD 노트는 길이가 있어 두 가지 처리가 들어간다:

1. **세로 스케일 확장**: `(durMs / previewMs) * (spawnY - judgmentY)` 만큼 늘림
2. **위치 오프셋**: 타일의 *중앙*이 아니라 **바닥**이 판정선에 닿도록 `+ holdHalfExtent` 보정

이전 구현은 중앙이 판정선에 도달했는데, 그러면 큰 HOLD 타일의 바닥이 화면 밖으로 미리 사라져 보이는 문제가 있었다(`NoteController.cs:104–116` 주석 참고).

#### 자동 미스

```csharp
if (!judged && songTime > hitTimeMs + missGraceMs)
{
    judged = true;
    onAutoMiss?.Invoke(this);
    // ...
    Destroy(gameObject);
}
```

`missGraceMs`는 난이도별 Good 윈도(Easy 225ms, Normal 180ms). 시작 탭이 이 범위를 넘으면 자동 미스로 처리.

HOLD 노트는 자동 미스 후에도 회색 처리하고 시각적으로 떨어지는 잔상을 끝까지 보여준다(`Destroy(gameObject, remainingSec)`). 큰 막대가 갑자기 사라지면 사용자가 당황하기 때문.

## 3. 판정

### 판정 윈도 (`JudgmentEvaluator`)

```
              Easy        Normal
Perfect       ±75ms       ±60ms
Great         ±150ms      ±120ms
Good          ±225ms      ±180ms
Miss          그 이상       그 이상
```

코드(`Assets/Scripts/Gameplay/JudgmentEvaluator.cs`):

```csharp
public static JudgmentResult Evaluate(int deltaMs, Difficulty difficulty)
{
    int abs = deltaMs < 0 ? -deltaMs : deltaMs;
    int perfect, great, good;
    if (difficulty == Difficulty.Easy) {
        perfect = 75; great = 150; good = 225;
    } else {
        perfect = 60; great = 120; good = 180;
    }
    if (abs <= perfect) return new JudgmentResult(Judgment.Perfect, deltaMs);
    if (abs <= great)   return new JudgmentResult(Judgment.Great, deltaMs);
    if (abs <= good)    return new JudgmentResult(Judgment.Good, deltaMs);
    return new JudgmentResult(Judgment.Miss, deltaMs);
}
```

`deltaMs`는 부호 있음 (음수 = 사용자가 일찍 탭, 양수 = 늦게 탭). `LatencyMeter`(디버그 빌드)가 이걸 사용해 사용자의 평균 편향을 보여준다.

### 매칭 알고리즘 (`JudgmentSystem.HandleTap`)

탭이 들어오면 같은 레인의 pending 노트 중 **abs(delta)가 가장 작은** 것을 고른다.

```csharp
foreach (var n in pending) {
    if (n.Lane != tapLane) continue;
    int delta = tapTimeMs - n.HitTimeMs;
    int abs = delta < 0 ? -delta : delta;
    if (abs < closestAbsDelta) {
        closestAbsDelta = abs;
        closest = n;
    }
}
```

**왜 첫 번째 노트가 아닌가**: 노트가 빠르게 연속될 때, 사용자가 두 번째 노트를 일찍 친 게 첫 번째 노트의 늦은 탭으로 잘못 판정되는 걸 방지.

### 판정 후 처리

| 결과 | 동작 |
|---|---|
| Perfect/Great/Good | `score.RegisterJudgment` + `OnJudgmentFeedback` 발화 + 노트 `MarkJudged()`/`MarkAcceptedAsHold()` |
| Miss | **점수에는 반영 안 됨** (note는 pending에 남아 자동 미스 대기), 그러나 피드백은 발화 — 사용자가 친 시도에 시각/햅틱 응답 |

이 "Miss는 score에 미반영" 분기는 `JudgmentSystem.cs:106` 주석에서 명확히 설명한다.

### 자동 미스 + 홀드 깨짐

```csharp
public void HandleAutoMiss(NoteController note) {
    pending.Remove(note);
    score.RegisterJudgment(Judgment.Miss);
    OnJudgmentFeedback?.Invoke(Judgment.Miss, note.transform.position);
}

public void HandleHoldBreak(NoteController brokenNote) {
    score.RegisterJudgment(Judgment.Miss);
    OnJudgmentFeedback?.Invoke(Judgment.Miss, brokenNote.transform.position);
}
```

## 4. 홀드 노트

### `HoldStateMachine` (`Assets/Scripts/Gameplay/HoldStateMachine.cs`)

순수 데이터 FSM (MonoBehaviour 아님). 4개 상태:

```
        OnStartTapAccepted
Spawned ───────────────────► Holding
                                │
                                ├── 판정선 종료 시각 도달 → Completed
                                │
                                └── 사용자가 레인 손 뗌 → Broken
```

`Tick(songTimeMs, pressedLanes, outTransitions)`을 매 프레임 호출 — `Holding` 상태의 entry들에 대해:
- 해당 레인이 `pressedLanes`에 없으면 → `Broken`
- 아니면 `songTimeMs >= endMs`이면 → `Completed`

### `HoldTracker` (`Assets/Scripts/Gameplay/HoldTracker.cs`)

`HoldStateMachine`을 Unity 씬에 연결하고 + **250ms마다 홀드 음 retrigger**를 담당한다.

#### 250ms 리트리거

```csharp
private const int HOLD_RETRIGGER_INTERVAL_MS = 250;
private const float HOLD_RETRIGGER_VOLUME = 0.7f;

// 매 프레임
foreach (var kv in holdAudio) {
    if (songMs - kv.Value.lastRetriggerMs < 250) continue;
    retriggerBuffer.Add(kv.Key);
    audioPool.PlayForPitch(kv.Value.pitch, 0.7f);
}
```

**왜 250ms인가**: 피아노 톤은 자연 감쇠가 빠르다. 긴 HOLD를 그냥 누르고만 있으면 음이 사라져 무음 상태가 된다. 250ms 간격으로 약하게(0.7) 재타격해서 sustain을 만든다.

#### 두 단계 버퍼링 패턴

```csharp
foreach (var kv in holdAudio) {
    // ... retriggerBuffer.Add(kv.Key)
}
for (int i = 0; i < retriggerBuffer.Count; i++) {
    int id = retriggerBuffer[i];
    var st = holdAudio[id];
    st.lastRetriggerMs = songMs;
    holdAudio[id] = st;  // 인덱서 set은 enumeration 후에
}
```

`Dictionary` enumeration 중에 인덱서로 값을 갱신하면 `InvalidOperationException`(2026-04-23 Unity 6000.3.13f1 Mono EditMode에서 확인). 두 단계로 분리.

#### `LaneGlowController` 연동

홀드 시작 시 `laneGlow.On(note.Lane)`, 종료/깨짐 시 `Off(note.Lane)`. 활성 레인은 사인파로 펄스 — `LaneGlowController`(`Feedback/`).

## 5. 오디오 동기 + 시계

### `AudioSyncManager` (`Assets/Scripts/Gameplay/AudioSyncManager.cs`)

곡 시계의 **권위 소스**. 다른 모든 컴포넌트는 `audioSync.SongTimeMs`만 본다.

#### 시계 계산

```csharp
public int SongTimeMs => GameTime.GetSongTimeMs(
    paused ? pauseStartDsp : TimeSource.DspTime,
    songStartDsp,
    CalibrationOffsetSec);

// GameTime.cs
public static int GetSongTimeMs(double nowDsp, double songStartDsp, double calibOffsetSec)
{
    double sec = nowDsp - songStartDsp - calibOffsetSec;
    return (int)(sec * 1000.0);
}
```

`scheduleLeadSec = 0.5`초의 버퍼: `StartSilentSong()` 호출 시점이 아니라 0.5초 후가 곡의 t=0이다. 오디오 스케줄러가 첫 음을 미리 준비할 시간 확보.

#### 일시정지

```csharp
public void Pause() {
    if (paused || !started) return;
    pauseStartDsp = TimeSource.DspTime;  // 정지 시각 기록
    AudioListener.pause = true;
    paused = true;
}

public void Resume() {
    if (!paused) return;
    double elapsed = TimeSource.DspTime - pauseStartDsp;
    songStartDsp += elapsed;  // ← 곡 시작 시각을 미룸 = SongTimeMs 동결
    AudioListener.pause = false;
    paused = false;
}
```

`SongTimeMs`는 일시정지 동안 동결되므로 `NoteController` 위치도 멈추고 `JudgmentSystem`의 delta 계산도 영향 안 받는다.

#### 테스트 seam (`ITimeSource`)

```csharp
public interface ITimeSource { double DspTime { get; } }

internal class RealTimeSource : ITimeSource {
    public double DspTime => AudioSettings.dspTime;
}

public ITimeSource TimeSource { get; set; } = new RealTimeSource();
```

EditMode 테스트는 수동 클럭(`ManualTimeSource { public double DspTime { get; set; } }`)을 주입해서 dspTime을 결정적으로 진행시킨다. 자세한 패턴은 [09-테스트-가이드.md](./09-테스트-가이드.md).

## 6. 다중 피치 샘플 풀

### `AudioSamplePool` (`Assets/Scripts/Gameplay/AudioSamplePool.cs`)

16채널 라운드로빈 `AudioSource` 풀 + MIDI 피치 → 가장 가까운 샘플 매핑 + 반음 단위 피치 시프트.

#### 채널 풀

```csharp
[SerializeField] private int channels = 16;
private AudioSource[] sources;
private int nextIndex;

public AudioSource NextSource() {
    var src = sources[nextIndex];
    nextIndex = (nextIndex + 1) % sources.Length;
    return src;
}
```

16채널인 이유: 빠른 곡에서도 동시 발음(이전 음의 감쇠 + 새 음)을 끊지 않으면서, 모바일 GPU/CPU 부담을 피하는 균형점.

#### 피치 → 샘플 매핑

```csharp
[SerializeField] private AudioClip[] pitchSamples;  // 17개
[SerializeField] private int baseMidi = 36;          // C2
[SerializeField] private int stepSemitones = 3;      // minor third

public static (AudioClip clip, float pitchRatio) ResolveSample(
    int midiPitch, AudioClip[] pitchSamples, int baseMidi, int stepSemitones)
{
    int hi = baseMidi + (pitchSamples.Length - 1) * stepSemitones;
    int p = Math.Clamp(midiPitch, baseMidi, hi);

    int baseIdx = (p - baseMidi) / stepSemitones;
    int sampleMidi = baseMidi + baseIdx * stepSemitones;
    int offset = p - sampleMidi;  // 0, 1, or 2

    // offset == 2일 땐 다음 샘플의 -1 시프트가 더 자연스러움
    if (offset == 2 && baseIdx + 1 < pitchSamples.Length) {
        baseIdx += 1;
        offset = -1;
    }

    float ratio = Mathf.Pow(2f, offset / 12f);  // 반음 = 2^(1/12)
    return (pitchSamples[baseIdx], ratio);
}
```

샘플은 3 반음 간격(C2, Ds2, Fs2, A2, C3, ...). 임의의 MIDI 피치 P에 대해:
- 가장 가까운 샘플 인덱스 = `(P - 36) / 3`
- 시프트량 = ±1 반음 이내(0, +1, -1)
- 피치 비율 = `2^(시프트/12)`

Unity `AudioSource.pitch`로 비율을 적용하면 샘플이 시간축으로도 늘어나지만(피치 ↑ = 빨라짐), 짧은 노트 주변 한두 반음 시프트는 청감상 자연스럽다.

### `LanePitches` (`Assets/Scripts/Gameplay/LanePitches.cs`)

채보가 피치를 안 줄 때(또는 사용자가 pending 노트 없는 곳을 탭할 때) 쓰는 기본값:

```
레인 0 → MIDI 48 (C3)
레인 1 → MIDI 55 (G3)
레인 2 → MIDI 60 (C4)
레인 3 → MIDI 67 (G4)
```

**왜 5도 계단인가**: 출시 4곡의 키(A-minor, D/C/Db-major)와 충돌이 적은 조합. 사용자가 빈 곳을 두드려도 음악적으로 어긋나지 않는다.

#### 만점 = 1,000,000

|---|---|

## 8. 캘리브레이션

### 흐름

첫 플레이 시 (`UserPrefs.HasCalibration == false`):

```
CalibrationController.Begin
   │
   ├── 사용자에게 "탭하세요" 표시
   │
   ▼ (Start 버튼)
8회 click을 0.5초 간격으로 PlayScheduled
   │
   ▼ (사용자 탭마다 dspTime 기록)
Evaluate()
   │
   ▼
CalibrationCalculator.Compute(expected[], tap[])
```

### `CalibrationCalculator.Compute`

```csharp
public static Result Compute(double[] expectedDspTimes, double[] tapDspTimes) {
    // 1. delta = tap[i] - expected[i] 페어 계산
    var deltas = new List<double>(pairCount);
    for (int i = 0; i < pairCount; i++)
        deltas.Add(tapDspTimes[i] - expectedDspTimes[i]);

    // 2. outlier 트림 (count >= 6일 때 첫·마지막 제거)
    deltas.Sort();
    if (deltas.Count >= 6) {
        deltas.RemoveAt(deltas.Count - 1);
        deltas.RemoveAt(0);
    }

    // 3. median 계산 → ms 단위 + ±500ms clamp
    double median = Median(deltas);
    int offsetMs = Mathf.RoundToInt((float)(median * 1000.0));
    offsetMs = Math.Clamp(offsetMs, -500, 500);

    // 4. MAD (Median Absolute Deviation) 계산
    var absDev = deltas.Select(d => Math.Abs(d - median)).OrderBy(x => x).ToList();
    double mad = Median(absDev);
    int madMs = Mathf.RoundToInt((float)(mad * 1000.0));

    // 5. 신뢰도: MAD <= 50ms이면 reliable
    return new Result { offsetMs, madMs, reliable = madMs <= 50 };
}
```

### 왜 median + MAD인가

- **median**: outlier에 강하다. 한 번 늦은 탭이 mean을 크게 왜곡하지만 median은 거의 영향 안 받는다.
- **MAD**: 분산(stddev)도 outlier에 약하다. MAD = `median(|x - median(x)|)`는 분포의 robust한 산포 척도.

### 신뢰도 처리

```
reliable (MAD ≤ 50ms) → 저장 + 종료
!reliable
   ├── retry < 3 → "다시 해보세요 (MAD XX ms)" + 재실행
   └── retry == 3 → 0ms 저장 + 종료 (포기)
```

3회 시도 후에도 안 되면 0ms로 저장하고 진행 — 사용자를 무한 캘리브레이션 루프에 가두지 않는다.

### 저장과 적용

```csharp
private void Save(int offsetMs) {
    UserPrefs.CalibrationOffsetMs = offsetMs;       // PlayerPrefs
    audioSync.CalibrationOffsetSec = offsetMs / 1000.0;  // 즉시 적용
}
```

`UserPrefs.CalibrationOffsetMs`는 `PlayerPrefs`로 저장되므로 앱 종료 후 다음 실행에서도 살아있다. `MigrateLegacy`가 W4 이전 키(`CalibOffsetMs`)를 신키로 마이그레이션.

### 재캘리브레이션

`SettingsScreen`의 "재캘리브레이션" 버튼이 `calibration.Begin(onDone: () => Show())` 호출 — 캘리브레이션 끝나면 Settings 오버레이가 다시 뜬다.
